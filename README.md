# House Prices - Advanced Regression Techniques

## კონკურსის მიმოხილვა 
House Prices კონკურსის მიზანია, დავაპროგნოზიროთ სახლის ღირებულება მისი ადგილმდებარეობის, 
მდგომარეობის, ოთახების რაოდენობისა თუ სხვა მისთ. მიხედვით. ამოცანის შეფასების კრიტერიუმია 
RMSE (Root-Mean-Squared-Error) პროგნოზის ლოგარითმსა და რეალური ღირებულების ლოგარითმს შორის. 

ჩემი მიდგომა გულისხმობს სრული ფაიფლაინის აგებას data cleaning, feature engineering, categorial
encoding, feature selection ნაბიჯებითა და საბოლოოდ წრფივი რეგრესიის მოდელის დატრენინგებას.

## EDA
დავყავი სვეტები კატეგორიულ და რიცხვით კატეგორიებად. 
დავაკვირდი სვეტებს nan-ების შემცველობის მიხედვით. 
ყურადსაღებია, რომ კატეგორიულ სვეტებში nan უმეტესწილად ინფორმაციის არარსებობას კი არ გულისხმობს, არამედ თავის თავში
ატარებს მნიშვნელოვან ინფორმაციას, როგორც, მაგალითად, PoolQC-ის შემთხვევაში- აქ nan ნიშნავს, 
რომ აუზი არ არსებობს და არა იმას, რომ ინფორმაცია არ გვაქვს აუზის ხარისხის შესახებ.

## Data Cleaning
სვეტები დავყავი სამი თვისების მიხედვით:
- nan-ის მნიშვნელობის;
- მონაცემთა ტიპის (კატეგორიული/რიცხვითი);
- ენკოდირების სტრატეგიის.
1. na_cat_cols: აქ nan feature-ის არარსებობას გულისხმობს. მაგალითად: არაა აუზი, არაა გარაჟი და სხვ.
2. mode_cat_cols: ყველა სხვა კატეგორიული სვეტი, რომელთა nan-ები მოდით ჩანაცვლდება.
3. target_cols: ნომინალური კატეგორიული სვეტები, რომლებზეც ჩატარდება Target Encoding.
4. ord_cols: ორდინალური კატეგორიული სვეტები, რომლებიც თავის თავში გარკვეულ დალაგებას, შეფასებას გულისხმობს. მაგალითად: ExterQual: Excellent-5, Good-4 და სხვ. 
5. drop_cols: Id

ამ საწყისი დაყოფის შემდეგ კიდევ ერთი დაყოფაა: 
1. target_na_cols: target_cols და na_cat_cols გადაკვეთა;
2. target_mode_cols: target_cols და mode_cat_cols გადაკვეთა;
3. ord_na_cols: ord_cols და na_cat_cols გადაკვეთა;
4. ord_mode_cols: ord_cols და mode_cat_cols გადაკვეთა;
5. num_cols: რიცხვითი სვეტები.

გამოვყავი ორდინალური სვეტებიდან ისინი, რომლებსაც ხარისხის ერთიანი სისტემა აქვთ და ისინი, 
რომლებსაც ინდივიდუალური მნიშვნელობები აქვთ. გავაკეთე შესაბამისი map და OrdinalMapper კლასი. 
შევქმენი შესაბამისი pipeline-ები ზემოაღნიშნული დაყოფის შესაბამისად. საბოლოოდ ყველა ფაიფლაინი
ერთიან ColumnTransfer-ში ჩავწერე.

## Feature Engineering
შევქმენი ახალი სვეტები:
1. TotalSF: სახლის მთლიანი ფართობი;
2. TotalBathPerArea: სახლში ყველა აბაზანის რაოდენობა ფართობის ერთეულზე;
3. HouseAge: სახლის ასაკი;
4. RemodAge: რამდენი წელი გავიდა სახლის რემონტიდან მის გაყიდვამდე;
5. RoomsPerArea: ოთახების რაოდენობა ფართობის ერთეულზე.

## Feature Selection
გამოვიყენე ორი მიდგომა:
1. Correlation Filter: ავაგე კორელაციის მატრიცა და თუკი feature-ების წყვილის კორელაცია გარკვეულ ზღვარზე მაღალია,
ამ წყვილიდან ვშლი იმას, რომელსაც საშუალოდ მეტი კორელაცია აქვს სხვა feature-ებთან. 
2. Recursive Feature Elimination (RFE): რეკურსიულად აშორებს ყველაზე ნაკლებად მნიშვნელოვან feature-ს. იტოვებს 
n_features_to_select რაოდენობის feature-ებს. 

## Training
1. Linear Regression: 

უპირველეს ყოვლისა, ვცდილობდი კატეგორიული სვეტების რიცხობრივად გადაკეთებას One-Hot Encoding-ის 
გამოყენებით, მაგრამ თითქმის 0.1-ით უკეთესი შედეგი საბოლოოდ Target Encoding-მა აჩვენა. 
ამის მიზეზი, ჩემი აზრით, შემდეგში მდგომარეობს: OHE-ს თითოეული სვეტი ორობით სვეტად გადაჰყავს და, 
შესაბამისად, არ ინახავს ინფორმაციას იმის შესახებ, თითოეული კატეგორიისთვის სახლის ფასი როგორია. 
Target Encoding კი სწორედ ამ უკანასკნელს ითვალისწინებს. შემდგომი ტრენინგი უფრო ეფექტურია, 
რადგან კატეგორიებში ინახება ინფორმაცია ფასთან მათი მიმართების შესახებ. თუმცა ისიც აღნიშნავია, 
რომ ამ მიდგომამ შეიძლება, data leakage გამოიწვიოს.
ისიცაა აღსანიშნავი, რომ OHE უკეთეს შედეგს დებდა ვალიდაციაზე, ვიდრე Target Encoding, მაგრამ პირიქით აღმოჩნდა
ტესტირების ეტაპზე. ამის ბრალი ისაა, რომ OHE უფრო სტაბილურია და ნაკლებად გამოიწვევს overfitting-ს, მაგრამ, 
კომპლექსური კავშირების შესწავლა უფრო უჭირს და ტესტზე underfitting-ს ვაწყდებით. Target Encoding-ს შეუძლია,
უფრო კომპლექსური კავშირები შეისწავლოს და უნახავ მონაცემებზე უკეთესი განზოგადება გამოსდის, თუმცა შესაძლოა, 
overfitting-შიც გადავიდეს. 

Feature Engineering-ის ეტაპზე, მას მერე, რაც შევქმენი ახალი სვეტები, გადავწყვიტე, ამ სვეტების 
შექმნაში გამოყენებული სვეტები წამეშალა, მაგრამ ამან უარესი შედეგი გამოიღო. 

კორელაციის ფილტრზე ვცადე threshold-ები 0.5-დან 0.95-მდე, თუმცა საუკეთესო შედეგი დადო 0.8-მ.
დაბალი მაჩვენებელი underfitting-ს იწვევს, რადგან იშლება მნიშვნელოვანი ინფორმაცია, მაღალი
მაჩვენებლის შემთხვევაში კი- overfitting-ს, რაც მოდელს უფრო მგრძნობიარეს ხდის ხმაურის მიმართ
და, შესაბამისად, ზრდის ვარიაციას. 

One-Hot Encoding-ის დროს თუკი RFE-ს ვამატებდი, უარესი შედეგი იდებოდა, ვიდრე მის გარეშე. OHE ძალიან ბევრ სვეტს 
ამატებს მაშინ, როდესაც ბევრი კატეგორიაა, მაგალითად, 20-ზე მეტს neighbourhood-ის შემთხვევაში. 
ამ დროს საწყისი სვეტის "მნიშვნელობა" ბევრ სხვადასხვა სვეტზე ნაწილდება და თითოეული ეს ახალი სვეტი შეიძლება, 
"უმნიშვნელოდ" გამოიყურებოდეს, რის გამოც RFE საჭირო ინფორმაციას შლიდა. 

Target Encoding თავის თავში ინახავს სვეტის პირდაპირ კავშირს target-თან და რაკი feature-ების განზომილება
არ იზრდება, თითოეულ feature-ზე მეტი "სიგნალი" მოდის, რის გამოც RFE ამ შემთხვევაში უკეთეს შედეგს დებს. 
ამ დროს RFE უკვე მართლაც "უმნიშვნელო" სვეტებს შლის. 

n_features_to_select-ის ასარჩევად ვცადე 5 ვარიანტი: 10, 20, 50, 65, 80. საუკეთესო შედეგი დაიდო 65-ის შემთხვევაში.
10, 20, 50-ის დროს underfitting-ია, რადგან მნიშვნელოვანი feature-ები იშლება, ხოლოდ 80-ის შემთხვევაში- ბევრი ხმაურიანი
feature რჩება, რაც overfitting-ს იწვევს. 

k-fold-მა ვალიდაციაზე უარესი შედეგი დადო, ვიდრე სტანდარტულმა ერთმა გაყოფმა, მაგრამ- უკეთესი შედეგი ტესტზე (დაახლოებით 
0.01-ით). 
ამის მიზეზი ვფიქრობ, ისაა, რომ მხოლოდ ერთი დაყოფის შემთხვევაში შესაძლოა, მოდელს "გაუმართლოს", ოპტიმისტური შედეგი დადოს.
k-fold დებს უფრო რეალურ შედეგს, შესაბამისად, მცირდება overfitting კონკრეტული დაყოფისადმი. უნახავ ტესტ დატაზე k-fold უკეთეს
შედეგს დებს. ისიც გასათვალისწინებელია, რომ არც ისე ბევრი მონაცემია საწყის დატაში- ესეც არგუმენტია იმისა, 
რატომაა k-fold ოპტიმალური. 

2. Decision Tree: 

საინტერესოა, შეძლებდა თუ არა არაწრფივი მოდელი უკეთესი შედეგის დადებას ვიდრე წრფივი რეგრესია
(სპოილერი: ვერა). ვცვლიდი შემდეგ ჰიპერპარამეტრებს: max_depth (3,5,10,None), min_samples_leaf (1,2,4,5), 
min_samples_split (2,5,10), criterion (squared_error, friedman_mse), max_features(None, sqrt, log2). 
აქ k-fold იმისთვის გამოვიყენე, ჰიპერპარამეტრთა თითოეული კომბინაციისთვის რომ დავრწმუნებულიყავი, კონსისტენტურ შედეგს
თუ ვიღებ. თანაც decision tree უფრო სენსიტიურია მონაცემთა მცირე ცვლილების მიმართ, ამიტომ k-fold აქ საუკეთესო
არჩევანია. 

როცა max_depth მცირეა, underfitting-ია, რადგან კომპლექსურ კავშირებს ვერ სწავლობს. სამაგიეროდ, როდესაც
ეს მნიშვნელობა დიდია, overfitting-ის შემთხვევაა. min_samples_split დაბალი მნიშვნელობა პატარა პატერნებს 
სწავლობს და overfitting-ის რისკია. უფრო მაღალი მნიშვნელობები ვარიაციას ამცირებს და ხის კომპლექსურობას თავიდან
გვარიდებს. min_samples_leaf დაბალი მნიშვნელობა overfitting-ის რისკია, მაღალ მნიშვნელობას განზოგადების
უკეთესი შანსი აქვს. model_max_features None ყველა feature-ს იყენებს, sqrt და log2 რანდომულად გარკვეულ ქვესიმრავლეს
ირჩევენ. 

საუკეთესო ჰიპერპარამეტრები აღმოჩნდა: max_depth = None, min_samples_leaf = 4, min_samples_split = 2, 
max_features = None, criterion = squared_error.
ე.ი. საუკეთესო შეფასება კომპლექსურმა ხემ, რომელსაც overfitting შეეზღუდა min_samples_leaf = 4 ჰიპერპარამეტრით.
თითოეულ split-ზე განიხილებოდა ყველა feature (max_features = None). ვვარაუდობ, იმიტომ, რომ პრეპროცესინგის დროს
უკვე საკმარისად გაიფილტრა და აირჩა საუკეთესო მონაცემები, შესაბამისად თითოეულის განხილვა გაყოფისას საუკეთესო
ვარიანტია. criterion = squared_error უფრო სტაბილურია, ვიდრე friedman_mse და უფრო უნივერსალურია. რაკი 
min_samples_leaf = 4 ამით უკვე შეიზღუდა ხე, min_samples_split = 2 საუკეთესოა, რადგან კომპლექსური კავშირის
